# 005. Use Conditional Writes for Item Creation

## Status

Accepted

## Context

The API supports item create, get, update, and delete operations backed by DynamoDB. Item IDs are generated by the create handler and are used as the DynamoDB partition key.

A simple DynamoDB `PutItem` can overwrite an existing item when the same `id` is written again. Even though UUID collisions are unlikely, overwrite behavior can hide duplicate creates or future changes that make IDs caller-provided or otherwise less isolated.

## Decision

Use a DynamoDB conditional write for item creation. The create handler uses `attribute_not_exists` on the item `id` so the write succeeds only when no item with that generated key already exists.

Map DynamoDB `ConditionalCheckFailedException` errors to a `409 Conflict` API response. Keep unexpected DynamoDB or internal errors mapped to safe `500` responses that do not expose raw internal error details to callers.

This prevents overwrite for a specific generated key. It does not make `POST /items` retry-idempotent because each retry can generate a new UUID and create a separate item. Idempotency keys or caller-provided request tokens remain deliberately out of scope for this project.

## Rationale

Conditional writes let DynamoDB enforce the no-overwrite rule at the storage layer, where concurrent requests for the same key are resolved consistently. This gives the API clearer create semantics: a successful create means a new item was stored for the generated key, while a duplicate key is reported as a conflict.

The handler still remains small, and the API response contract stays focused on observable behavior rather than SDK command internals.

## Trade-Offs

- Slightly more handler logic is needed to distinguish conditional failures from unexpected errors.
- API semantics are clearer because duplicate creates no longer look like successful overwrites.
- Accidental overwrites are better protected when the same key is written more than once.
- Retry idempotency is not provided; clients that retry `POST /items` after an uncertain outcome may create more than one item.
- The design more directly acknowledges DynamoDB's conditional-write and consistency model instead of relying only on low collision probability.

## Test Coverage

Handler tests cover successful item creation returning `201`, duplicate item creation returning a safe `409`, and unexpected DynamoDB failures returning a safe `500`. These tests assert the public API response behavior rather than tightly coupling the suite to AWS SDK command internals.

## Consequences

Create behavior is more explicit without changing the table design, adding services, or changing the public route shape. If item IDs later become caller-provided, the same conditional-write behavior remains important and should stay covered by tests. If retry idempotency becomes a requirement, the API should add an explicit idempotency-key design instead of treating generated UUID creates as idempotent.
